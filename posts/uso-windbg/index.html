Explicación de comandos en WinDBG

Nov 7, 2025
#WinDBG

WinDbg es un depurador que se puede usar para analizar volcados de memoria, depurar el modo de usuario activo y el código en modo kernel, y examinar los registros de CPU y la memoria.

En este caso solo nos enfocaremos en el modo usuario, y se mostrarán los ejemplos con el bloc de notas de Windows de 32 bits (notepad).

Interfaz de WinDbg

La pantalla de inicio del depurador es la siguiente:

Para "adjuntar" un proceso, haremos clic en archivo situado en la parte izquierda superior y como se muestra en la foto, de nuevo, haremos doble clic sobre notepad.exe.

El depurador inyecta un punto de interrupción de software sobrescribiendo la instrucción actual en la memoria con una instrucción de ensamblador (INT 3).

WinDbg también cuenta con símbolos de depuración que le permiten referenciar funciones internas, estructuras y variables globales mediante nombres en lugar de direcciones.

Los archivos .PDB, se crean cuando Microsoft compila los archivos nativos en Windows. Microsoft no proporciona estos archivos de símbolos para todos los archivos, además las aplicaciones de terceros pueden tener sus propios archivos de símbolos.

La ruta que se utiliza es C:\Symbols y los módulos se pueden cargar mediante .reload.

0:007> .reload /f
Reloading current modules
...

Escritura y búsqueda en la memoria

Los mismo modificadores que se utilizan para el comando de visualización también se aplica para los de edición.

Se muestra un ejemplo modificando un DWORD que apunta a ESP:

0:003> dd esp L1
02dbf8f8  775bb509

0:003> ed esp 42424242

0:003> dd esp L1
02dbf8f8  42424242

Al igual que con los comandos de visualización también podemos modificar directamente caracteres ASCII y unicode con ea y eu respectivamente:

0:003> dd esp L2
02dbf8f8  41414141 0c027ab8

0:003> ea esp "Hola"

0:003> dc esp L2
02dbf8f8  616c6f48 0c027ab8                    Hola.z..

Para poder buscar en la memoria se usa el comando s.

Modificaremos el registro ESP para luego utilizar el comando de búsqueda:

0:007> ed esp 41414141

0:007> dd esp L2
061afc3c  41414141 ab95988b

Para realizar una búsqueda necesitamos lo siguiente:

  • El tipo de memoria a buscar.
  • El punto de inicio de la memoria a buscar.
  • La longitud de la memoria a buscar.
  • El patrón a buscar.

Al buscar un tipo de memoria dword, usaremos d, establecemos la longitud de búsqueda en 0, ya que queremos que busque desde el principio y que abarque todo el espacio de la memoria que se representa con ?80000000, en este caso se buscará 41414141.

0:007> s -d 0 L?80000000 41414141
061afc3c  41414141 ab95988b 775bb4d0 775bb4d0  AAAA......[w..[w

Puntos de interrupción

Los puntos de interrupción se usan para detener la ejecución en ubicaciones de código deseadas.

Los puntos de interrupción controlados directamente por el depurador se conocen como puntos de interrupción de software.

Los puntos de interrupción controlados por el procesador y configurados a través del depurador se conocen como puntos de interrupción de hardware.

Puntos de interrupción de Software

Al colocar un punto de interrupción WinDbg reemplaza temporalmente el código de operación de la instrucción donde queremos que se detenga la ejecución con una instrucción de ensamblador (INT 3).

Como seguimos con el ejemplo del notepad, vamos a poner un punto de interrupción en la API de Windows (WriteFile) que se utiliza para escribir datos en un archivo específico o un dispositivo de entrada/salida.

bp kernel32!WriteFile

0:007> bl
0 e Disable Clear  769ed830     0001 (0001)  0:**** KERNEL32!WriteFile

En estos momentos está activado, cuando intentemos guardar el documento, habremos llegado al punto de interrupción:

Breakpoint 0 hit

eax=02bdebd0 ebx=00000000 ecx=00000006 edx=00000000 esi=00000006 edi=08504000
eip=769ed830 esp=02bdebb0 ebp=02bdebe4 iopl=0         nv up ei pl nz na pe nc
cs=001b  ss=0023  ds=0023  es=0023  fs=003b  gs=0000             efl=00200206
KERNEL32!WriteFile:
769ed830 ff252495a476    jmp     dword ptr [KERNEL32!_imp__WriteFile (76a49524)] ds:0023:76a49524={KERNELBASE!WriteFile (75558f20)}

Para habilitar y deshabilitar los puntos de interrupción, es de la siguiente forma:

0:000> bd 0 // deshabilitar

0:000> bl 
     0 d Enable Clear  769ed830     0001 (0001)  0:**** KERNEL32!WriteFile
    
0:000> be 0 // habilitar

0:000> bl
     0 e Disable Clear  769ed830     0001 (0001)  0:**** KERNEL32!WriteFile

Para borrarlo por completo es:

0:007> bl
     0 e Disable Clear  769ed830     0001 (0001)  0:**** KERNEL32!WriteFile
     
0:007> bc 0

0:007> bl
// nothing

También existen los puntos de interrupción no resueltos que se usan mediante bu, las diferencias entre uno resuelto y no resuelto es al siguiente:

Puntos de interrupción resueltos: Son aquellos puntos de interrupción que el depurador ha localizado con éxito en el código fuente y que están listos para ser utilizados durante la depuración.

Puntos de interrupción no resueltos: Son aquellos puntos de interrupción que el depurador no puede localizar en el código fuente.

  • Causas:
    • La línea de código especificada no existe en el archivo. - El depurador no tiene acceso al archivo o al código fuente. - Cambios en el código fuente después de establecer el punto de interrupción (por ejemplo, la línea se movió o se eliminó). - Problemas con el proceso de compilación o con el entorno de depuración.

Para poder cargarlos primero hay que cargar la lista de módulos lm m ole32:

0:007> bu OLE32!WriteStringStream

0:007> bl

Después de reanudar al ejecución observamos que se ha cargado OLE32.dll:

0:007>
ModLoad: 77400000 7743000 C:\Windows\System32\ole32.dll

También es importante recalcar que aunque el punto de interrupción se resuelva al cargar el dll, no se activo debido a que no se forzó una llamada a ole32!WriteStringStream.

Puntos de interrupción de Hardware

Los puntos de interrupción del hardware o del procesador son manejados por el procesador y almacenados en los registros de depuración del procesador. Pueden detener la ejecución del código cuando se realiza un tipo particular de acceso, como lectura, escritura o ejecución, a una ubicación de memoria específica.

La principal ventaja de los puntos de interrupción de hardware es que permiten supervisar los cambios o acceder a los datos en memoria.

En el siguiente ejemplo, vamos a establecer un punto de interrupción de hardware en la ejecución de WriteFile.

El resultado es equivalente a establecer un punto de interrupción de software, pero en este caso, aprovechamos la CPU y los registros de depuración, en lugar de modificar el código con una instrucción INT 3.

// 1 = Solo un byte

ba e 1 kernel32!WriteFile;g

Al guardar el archivo llegamos al punto de interrupción:

cs=001b  ss=0023  ds=0023  es=0023  fs=003b  gs=0000             efl=00200202
KERNEL32!WriteFile:
7629d830 ff2524952f76    jmp     dword ptr [KERNEL32!_imp__WriteFile (762f9524)] ds:0023:762f9524={KERNELBASE!WriteFile (756f8f20)}
